# M1 Progress Note - 2026-03-30

`novaaa64` now crosses the real UEFI boundary and reaches the full QEMU handoff chain.

This note captures the live Spark lane. The roadmap has since widened into the portable-fabric direction, so this file should be read together with `docs/roadmap/master-roadmap-checklist.md` and `docs/boot/bootinfo-v2.md`.

## What It Does Now

- boots as an AA64 UEFI application in QEMU
- uses the shared `libs/nova_rt` BootInfo ABI instead of a private loader copy
- records firmware revision and vendor pointer
- records config-table base/count
- records ACPI, SMBIOS, and DTB pointers when present
- records pre-exit memory-map entry count and descriptor size
- loads `stage1.bin`, `kernel.bin`, and `init.capsule` from the boot volume
- validates Nova Payload v1 headers and body digests for `stage1.bin` and `kernel.bin`
- records an explicit payload load window and entry ABI in `stage1.bin` and `kernel.bin`
- records explicit BootInfo presence flags for kernel digest and verification metadata
- defines a persistent `NovaVerificationInfoV1` contract describing stage1/kernel payload validation state
- stages real freestanding `stage1.bin` and `kernel.bin` payload images instead of Rust archive files
- reserves explicit loader-owned storage for BootInfo, the final UEFI memory map, and `Stage1Plan`
- computes a persistent SHA-256 digest object for the staged kernel image and exposes it through `kernel_image_hash_ptr`
- copies payloads and loader log into explicit `LOADER_DATA` pages
- prints the resulting handoff summary before leaving boot services
- performs `ExitBootServices()`
- refreshes BootInfo with the final post-exit memory-map buffer stored in reserved loader pages
- builds a real `novaos-stage1` handoff plan after boot services are gone and writes it into reserved loader storage
- requires the staged verification object before stage1 plan construction
- transfers into the loaded `stage1.bin` image with a persistent `Stage1Plan` pointer
- routes stage1 into kernel entry with the shared `NovaBootInfoV1` pointer ABI
- in QEMU, reaches `NovaOS stage0 post-exit`, `NovaOS stage1 entered`, and `NovaOS kernel entered`
- executes a real kernel-side raw-entry bring-up path that validates `NovaBootInfoV1`, probes the reserved memory-map window, and builds the early exception, page-table, and frame-allocator plans
- exposes kernel-image integrity metadata to the early kernel through the shared boot contract
- now re-enters the shared kernel runtime after raw validation and exposes early boot-console output over QEMU trace plus framebuffer when present
- keeps the current executable-image model explicit but limited to flat-binary payloads for now
- keeps Spark as the first truth platform while preserving the path to later workstation and server ports
- coexists with the new portability scaffolding in `libs/nova_fabric`, `services/acceld`, `services/memd`, `kernel/arch/x86_64`, and `drivers/bus/pci`

## Current Gap To M1 Acceptance

This is stronger than the earlier loader scaffold, but it is still not full M1 acceptance.

Remaining work:
- keep shrinking the raw-entry shim until the loader handles executable images more explicitly
- wire the richer shared kernel runtime more directly to the verification-object contract once that executable-image path is fully explicit
- grow the payload contract beyond the current flat-binary-only model when segment-aware loading becomes necessary
- prove the same post-exit chain on real Spark hardware
- carry the current Spark facts into a transitional BootInfo v2-compatible model before deeper platform-specific work lands
